# better-bucket v0.1.0 — Scaffold

**The repository bootstrap.** v0.1.0 stands the project up end to end: a fully
specified `Cargo.toml`, the crate root with its REPS lint posture, the test and
benchmark harnesses wired and running, the CI matrix green across all three
platforms, and the documentation shaped for the API that lands across the `0.x`
series. There is no token-bucket logic yet — by design. This release exists to
make every quality gate the implementation will be held to live and enforced
*before* the first line of accounting code is written.

## What is better-bucket?

A genuinely better token bucket for Rust. The hot path — `try_acquire` — is
designed to be lock-free, allocation-free, and cache-aligned: a single
compare-and-swap over a packed `(tokens, last_refill_tick)` word, with refill
computed lazily from a monotonic clock so an idle bucket costs nothing. The
defining correctness property is that the bucket never over-grants — across any
concurrent interleaving, total tokens handed out never exceed capacity plus
accrued refill. It is the single-purpose home for token-bucket math in the
wider ecosystem and the primitive the `rate-net` rate limiter consumes.

## What's in 0.1.0

### Crate manifest and feature surface

`Cargo.toml` carries the full published-crate metadata: Rust 2024 edition, MSRV
1.85, dual `Apache-2.0 OR MIT` license, keywords, categories, repository /
homepage / docs links, a `docs.rs` configuration, and a release profile tuned
for size and speed (`lto = "fat"`, `codegen-units = 1`, `panic = "abort"`,
`strip = "symbols"`).

Two feature flags, both on by default:

- **`std`** — the standard library. Turning it off makes the crate `no_std`; the
  caller then drives time explicitly. When `std` is on it propagates `std` to
  `clock-lib` via the weak `clock-lib?/std` dependency feature.
- **`clock`** — the pluggable [`clock-lib`](https://crates.io/crates/clock-lib)
  time source: a monotonic clock plus a mockable clock for deterministic,
  sleep-free tests.

A pure `no_std`, caller-driven-tick build drops `clock-lib` entirely:

```toml
better-bucket = { version = "0.1", default-features = false }
```

### Crate root and the `VERSION` constant

`src/lib.rs` establishes the REPS lint posture at the crate root —
`deny(warnings)`, `deny(missing_docs)`, and the `clippy` restriction set
(`unwrap_used`, `expect_used`, `print_stdout`, `dbg_macro`, …) — and wires
`no_std` on demand while keeping `std` available under `cfg(test)` so the unit
harness always links. The one item it exports is the version constant:

```rust
// Always populated; matches the linked crate's semantic version.
println!("better-bucket {}", better_bucket::VERSION);
assert!(better_bucket::VERSION.starts_with("0.1"));
```

### Test and benchmark harnesses

The full test stack is wired and exercised, not merely declared:

- **Criterion** (`benches/bucket_bench.rs`, `harness = false`) — a baseline for
  the packed-atomic `compare_exchange_weak` update the acquire path is built on,
  so there is a measured floor to track the real `try_acquire` against in 0.3.0.
- **`loom`** (`tests/loom_acquire.rs`, gated on `cfg(loom)`) — a model check
  that drains a shared budget from two racing threads via CAS and asserts it
  never over-grants. This is the harness the full acquire/refill interleaving
  proof replaces in 0.3.0; standing it up now confirms the model checker builds
  and runs against this crate's toolchain.
- **`proptest`** — declared in the dev-dependency stack, ready for the invariant
  suite that lands with the foundation API in 0.2.0.

The `loom` cfg (and the docs.rs `docsrs` cfg) are registered through
`[lints.rust]` `check-cfg`, so the gated test compiles cleanly under
`-D warnings` in normal builds.

### CI and the quality gates

`.github/workflows/ci.yml` runs three jobs:

- **Test matrix** — Linux, macOS, and Windows × stable and MSRV 1.85: format
  check, `clippy --all-targets --all-features -D warnings`, `test
  --all-features`, a `no_std` `test --no-default-features`, and `doc` under
  `RUSTDOCFLAGS=-D warnings`.
- **Loom** — `RUSTFLAGS="--cfg loom" cargo test --test loom_acquire`.
- **Security** — `cargo audit` against the RustSec advisory database and
  `cargo deny check` against the licenses / bans / sources / advisories policy
  in `deny.toml`.

## Breaking changes

**None.** This is the first release.

## Verification

Run on Windows x86_64 and on Linux x86_64 (WSL2 Ubuntu), Rust stable 1.95.x;
the MSRV 1.85 build is enforced by the CI matrix. Every command below passed on
both platforms:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo clippy --all-targets --no-default-features -- -D warnings
cargo test --all-features
cargo test --no-default-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
RUSTFLAGS="--cfg loom" cargo test --test loom_acquire
cargo deny check
cargo audit
```

Results: `1` unit test, `1` doctest, and `1` `loom` model check pass; the
`no_std` build is identical. `cargo deny check` reports advisories, bans,
licenses, and sources all ok; `cargo audit` scans 135 dependencies with no
known vulnerabilities.

## What's next

- **0.2.0 — Foundation.** Lock the public shape: `Bucket`, `BucketConfig`, the
  acquire-outcome type, the `error-forge` error type, and the `TokenBucket`
  trait. A simple, correct (not yet lock-free) implementation behind the Tier-1
  entry points, the no-over-grant invariant written down as a `proptest` suite,
  and a `MockClock`-driven refill test.
- **0.3.0 — The lock-free core.** Replace the simple implementation with the
  packed-atomic CAS acquire path, the `loom` interleaving proof, and the
  contended stress test.
