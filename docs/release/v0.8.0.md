# better-bucket v0.8.0 — Alpha (Consumer-Surface Validated)

**Used as a consumer would use it — and nothing got in the way.** v0.8.0 is the
alpha checkpoint: the public surface is exercised exactly the way a downstream
limiter such as `rate-net` will exercise it, and the result is that no API change
was needed. The crate is consumable as-is.

## What is better-bucket?

A genuinely better token bucket for Rust: lock-free, allocation-free, never
over-grants, API frozen. Token bucket is its sole job; it is the primitive
`rate-net` consumes.

## What's new in 0.8.0

### First-consumer shake-out

The roadmap's alpha phase is about putting the crate in front of its first real
consumer and fixing whatever that surfaces. `rate-net` is not built yet, so this
release does the next-best, honest thing: it **stands in as the consumer** and
drives the public surface the way `rate-net` will, in `tests/consumer_pattern.rs`:

- **Code against the trait, not the type.** A consumer's `check` takes
  `&dyn TokenBucket` and turns a `Decision` into its own allow/deny with a retry
  hint. Reads naturally.
- **Heterogeneous clocks behind one trait object.** A `Bucket<SystemClock>` (for
  production) and a `Bucket<Arc<ManualClock>>` (for tests) are both used through
  the same `&dyn TokenBucket`.
- **Per-key buckets sharing one clock.** Several buckets share a single injected
  `ManualClock`, so one `advance` refills every key's limiter deterministically —
  the pattern a keyed limiter's test suite needs.
- **Configured construction then clock injection.** `BucketConfig::new(..)` →
  `Bucket::from_config(..)` → `.with_clock(..)`, the path a consumer uses for
  non-default quotas.

All four pass, and — the point of the exercise — they read cleanly. The
`TokenBucket` trait, the clock-injection seam, and the `Decision` outcome cover
the keyed-limiter use case without friction.

### No API change

Nothing was added or altered. The frozen surface from `0.7` is sufficient for
the consumer pattern, so it stays exactly as it is. The keyed store itself — the
sharded per-key map, eviction, quotas — remains the consumer's concern, as the
design has always intended; this crate owns the token-bucket algorithm and stops
there.

## Breaking changes

**None.** Tests only.

## A note on the real integration

The literal integration against `rate-net` happens inside that crate, when it is
built, and must not reach back into this one. This release establishes that the
surface is ready for it: a consumer that codes against `TokenBucket`, injects a
clock, and reads `Decision` has everything it needs.

## Verification

Run on Windows x86_64 and on Linux x86_64 (WSL2 Ubuntu), Rust stable 1.95.x and
MSRV 1.85; the MSRV build is also enforced by the CI matrix.

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --no-default-features -- -D warnings
cargo test
cargo test --no-default-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps
RUSTFLAGS="--cfg loom" cargo test --test loom_acquire
cargo deny check
cargo audit
```

Counts at this tag (default features): 34 unit tests + 1 allocation audit + 13
hardening tests + 4 consumer-pattern tests + 3 property tests + 27 doctests;
`loom`: 1 model check. `cargo deny check` and `cargo audit` are clean.
