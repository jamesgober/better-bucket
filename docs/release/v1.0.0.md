# better-bucket v1.0.0 — Stable

**The freeze.** The public API is committed under strict SemVer: pin
`better-bucket = "1"` and trust that `1.x` stays backwards compatible. There are
no new features here — `1.0.0` is the stability commitment, plus a final
perfection pass over the code, the tests, and the docs.

## What is better-bucket?

A genuinely better token bucket for Rust. The hot path — `try_acquire` — is
lock-free (one `compare_exchange_weak` on a packed atomic word),
allocation-free, and cache-line aligned, with a division-free fixed-point refill
computed lazily from an injected monotonic clock. It never over-grants, and that
contract is proven, not asserted. Token bucket is its sole job; it is the
primitive `rate-net` consumes.

## The stability promise

[`docs/STABILITY.md`](https://github.com/jamesgober/better-bucket/blob/main/docs/STABILITY.md)
is the contract. In short:

- **Strict SemVer.** The frozen surface — `Bucket`, `BucketBuilder`,
  `BucketConfig`, `Decision`, `BucketError`, the `TokenBucket` trait, and
  `VERSION` — cannot change signature without a `2.0`. Additions are minor.
- **MSRV is Rust 1.85, frozen at 1.0.**
- **Behavioural contracts** are committed and tested: never over-grants, no
  panics, allocation-free acquire, an honest `retry_after`, validated
  construction, and the documented representation limits (the ~4.29 M-token
  capacity ceiling, millisecond `retry_after` resolution, and the wrap-safe
  clock).
- **Explicitly not promised:** `Display`/`Debug` text, internal timings, the
  transitive dependency tree, `#[doc(hidden)]` items, and the benchmark-only
  `comparison` feature.

## The 1.0 perfection pass

The `0.x` cycle built and hardened the crate; `1.0` locks it and tightens the
last details.

- **Hot path.** The contended CAS retry now emits a `spin_loop` hint to ease
  cache-line pressure (the uncontended path, which never retries, is unchanged),
  and the public methods are `#[inline]` so the acquire path inlines into
  consumer code. The bucket's own accounting measures **~5 ns**.
- **Accuracy certification.** A 16-thread, multi-token, exact-conservation test
  joins the suite: with a capacity that is not a multiple of the take size and
  no refill, every token must end up either granted or still available — none
  lost, duplicated, or granted beyond capacity. It holds run after run, on top
  of the `loom` model check, the allocation audit, the adversarial/edge suite,
  and the `proptest` invariants.
- **Honest benchmarks.** `docs/BENCHMARKS.md` carries the final numbers and a
  straight comparison with `governor` (see below).
- **Docs.** Pre-1.0 status notes are gone, every public item in `docs/API.md`
  carries multiple examples, and a `docs/DESIGN.md` and `docs/STABILITY.md`
  round out the reference.

## Performance, and an honest word on `governor`

On an AMD Ryzen 9 9950X3D (full numbers and method in `docs/BENCHMARKS.md`):

- The bucket's own accounting is **~5 ns**; a real `try_acquire` is **~21 ns**,
  the difference being the `Instant::now()` read, which dominates.
- Against `governor` on the same clock, `governor` is ~10–13% faster per call.
  That is expected and worth stating plainly: `governor` is **GCRA** (store one
  timestamp, compare it), while `better-bucket` is a **token bucket** (track and
  refill a count, clamp to a capacity, support multi-token acquires, bursts, an
  `available()` snapshot, and a `retry_after` hint). GCRA does less work.

So `better-bucket` does not claim to out-cycle a GCRA. It is the fastest, safest
**token bucket** in its class — lock-free and allocation-free where
`leaky-bucket` uses a background task and hand-rolled buckets use a `Mutex`,
with a `loom`-proven no-over-grant contract and real token semantics. Reach for
`governor` when a bare allow/deny is all you need; reach for `better-bucket` when
you want an actual token bucket. End-to-end, the clock bounds both.

## Breaking changes

**None.** Every `0.9.5` call site compiles and behaves identically.

## Verification

Run on Windows x86_64 and on Linux x86_64 (WSL2 Ubuntu), Rust stable 1.95.x and
MSRV 1.85; the MSRV build is also enforced by the CI matrix. All green:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --no-default-features -- -D warnings
cargo test
cargo test --no-default-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps
RUSTFLAGS="--cfg loom" cargo test --test loom_acquire
cargo bench --features comparison
cargo deny check
cargo audit
```

Counts at this tag (default features): 35 unit tests + 1 allocation audit + 13
hardening tests + 4 consumer-pattern tests + 4 property tests + 27 doctests;
`loom`: 1 model check. `cargo deny check` and `cargo audit` are clean.
