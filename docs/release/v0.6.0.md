# better-bucket v0.6.0 — Optimization

**Faster, division-free, and measured against the incumbent — honestly.** v0.6.0
takes the acquire path off the divider and onto a precomputed multiply-and-shift,
shaving ~9% off the single-thread hot path, and commits the head-to-head against
`governor`. The result is not the clean win the roadmap hoped for, and this note
says so plainly: on the same clock the two are tied, the bucket's *algorithm* is
at least as lean as `governor`'s, and `governor` wins out-of-the-box only because
its default clock is faster than the one `clock-lib` gives us. No API changed.

## What is better-bucket?

A genuinely better token bucket for Rust: one line to construct, one to check,
lock-free and allocation-free on the hot path, never over-grants. Token bucket is
its sole job; it is the primitive `rate-net` consumes.

## What's new in 0.6.0

### A division-free acquire path

The `0.5` refill computed `elapsed_ms * refill_amount * 1e9 / period_nanos` with
a 128-bit division on **every** acquire. v0.6 precomputes the refill rate once at
construction as a `Q22` fixed-point millitokens-per-millisecond multiplier, so
the hot path is a multiply and a shift — no division. A further early return
skips the arithmetic entirely when no whole millisecond has elapsed since the
last refill, which is the common case when acquires arrive in bursts.

The behaviour is unchanged (the fixed-point result is exact for the rates the
test suite pins, and the property tests still hold); only the cost moved. The one
division now happens at construction.

### The numbers (Ryzen 9 9950X3D, WSL2)

| Benchmark | median | vs 0.5 |
|---|---|---|
| `try_acquire` single-thread (real clock) | 24.1 ns | ~9% faster (was 26.5) |
| `try_acquire` algorithm only (mock clock) | 6.2 ns | — |
| contended, 8 threads | 4.4 ns | — |
| `available` refill-after-idle | 4.7 ns | flat |

The bucket's own accounting is **~6 ns**; the single-thread end-to-end figure is
dominated by the `Instant::now()` read, not the bucket.

### Head-to-head vs `governor`, in full honesty

| Limiter | clock | median |
|---|---|---|
| `better-bucket` | `Instant` (clock-lib) | 24.0 ns |
| `governor` | `Instant` (monotonic) | 23.2 ns |
| `governor` | `quanta` (default) | 7.1 ns |

- **On the same `Instant` clock the two are tied** (24.0 vs 23.2 ns, within a
  nanosecond, both clock-bound). `better-bucket` does not measurably beat
  `governor` here.
- **The algorithm is at least as lean.** `better-bucket` with a cheap clock
  (~6.2 ns) edges `governor` on its fast `quanta` clock (~7.1 ns).
- **Out of the box `governor` is faster end-to-end** (7 ns vs 24 ns), entirely
  because its default `quanta` clock beats the `Instant` clock `better-bucket`
  reads through `clock-lib`. A clock difference, not an algorithm one.

The takeaway: `better-bucket`'s end-to-end latency is bounded by the monotonic
clock, not its own code. Matching `governor` out-of-the-box needs a faster
monotonic source (e.g. TSC) from `clock-lib` — a cross-crate improvement noted
for `clock-lib`, not worked around here. Full method and numbers are in
[`docs/BENCHMARKS.md`](BENCHMARKS.md).

### SIMD: evaluated, declined

A single-bucket acquire is one CAS on one word — there is no lane-parallelism for
SIMD to exploit, and no consumer needs a batched multi-bucket acquire
(`rate-net` acquires one bucket at a time). Per the roadmap's "evaluate, don't
force" rule, SIMD is declined; it will be revisited only if a real batched
consumer path appears.

## Breaking changes

**None.** The optimization is internal; every signature is unchanged.

The benchmark-only `comparison` feature was added (it pulls `governor` for the
comparison bench). It is excluded from the default build, from CI, from the MSRV
matrix, and from docs.rs, so it does not affect the crate's dependency footprint
or supported Rust version. CI now tests the default and `no_std` feature sets
explicitly rather than `--all-features`, to keep the benchmark-only dependency
out of those builds.

## Verification

Run on Windows x86_64 and on Linux x86_64 (WSL2 Ubuntu), Rust stable 1.95.x and
MSRV 1.85; the MSRV build is also enforced by the CI matrix.

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --no-default-features -- -D warnings
cargo clippy --all-targets --features comparison -- -D warnings   # bench-only
cargo test
cargo test --no-default-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps
RUSTFLAGS="--cfg loom" cargo test --test loom_acquire
cargo bench --features comparison
cargo deny check
cargo audit
```

Counts at this tag (default features): 34 unit tests + 1 allocation audit + 3
property tests + 27 doctests; `loom`: 1 model check. `cargo deny check` reports
advisories, bans, licenses, and sources all ok; `cargo audit` is clean.
