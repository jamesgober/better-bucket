# better-bucket v0.3.0 — The Lock-Free Core

**The reason the crate exists.** v0.3.0 replaces the foundation's mutex with the
real thing: all mutable state now lives in a single `AtomicU64`, and
`try_acquire` is one `compare_exchange_weak` loop with lazy refill computed from
the injected monotonic clock. No lock, no allocation, cache-line aligned. The
no-over-grant contract is no longer asserted by hand-waving — it is proven by a
`loom` model check, an eight-thread stress test, and an allocation audit, on top
of the property suite carried forward from v0.2. The public surface did not
change: the same `Bucket`, `BucketConfig`, `Decision`, `BucketError`, and
`TokenBucket` from v0.2 now sit on a lock-free core.

## What is better-bucket?

A genuinely better token bucket for Rust. The common case is one line —
`Bucket::per_second(100)` then `bucket.try_acquire(1)` — and that one-line path
is the fast path. Refill is lazy: tokens accrue from a monotonic clock the
moment you ask, with no background timer thread. The defining correctness
property is that the bucket never over-grants. It is the single-purpose home for
token-bucket math in the wider ecosystem and the primitive the `rate-net` rate
limiter consumes.

## What's new in 0.3.0

### The lock-free acquire path

State is one `AtomicU64`:

- **upper 32 bits** — tokens in millitokens (thousandths of a token), for
  sub-token refill resolution;
- **lower 32 bits** — milliseconds since the bucket was created, marking the
  last refill computation.

`try_acquire(n)` reads the clock once, then runs a CAS loop: load the word,
compute the lazily-refilled token count, and — if `n` tokens are available —
`compare_exchange_weak` the new `(tokens − n, now)` into place. A denied request
writes nothing, so it never contends with a granting one. The bucket is
`#[repr(align(64))]`, so independent buckets never share a cache line.

Refill is computed from the refill period in **nanoseconds**, so a
sub-millisecond period (e.g. 5 tokens per 200µs) still accrues correctly even
though the time field ticks in milliseconds.

### `reset()`

```rust
use better_bucket::Bucket;

let bucket = Bucket::per_second(4);
assert!(bucket.try_acquire(4));
bucket.reset();              // back to full
assert_eq!(bucket.available(), 4);
```

A new method that refills to full and re-anchors the internal clock. It is the
escape hatch for the two packing limits below: discard accumulated debt for a
fresh burst, and keep refill alive on a process that outlives the millisecond
counter's saturation window.

### Proven, not asserted

The no-over-grant invariant is defended three new ways, all run in CI:

- **`loom`** (`tests/loom_acquire.rs`) drives the *real* `Bucket` from two
  threads whose combined demand exceeds supply, and exhaustively checks every
  interleaving grants exactly the available tokens — no over-grant, no lost
  token. Under `--cfg loom` the library's atomics switch to `loom::sync::atomic`
  so the model checker sees them.
- **A stress test** runs eight threads against one bucket with no refill and
  asserts exactly the capacity is granted.
- **An allocation audit** (`tests/alloc.rs`) installs a counting global
  allocator and asserts a 100,000-iteration acquire loop allocates zero times.

The v0.2 `proptest` invariants (tokens within `[0, capacity]`; grants never
exceed initial fill plus accrued refill) continue to pass against the new core.

## Breaking changes

**None to the public API** — every signature is unchanged from v0.2. Two
behavioural notes introduced by the packed representation:

- **Capacity is effectively capped at ~4.29 million tokens** (`u32::MAX`
  millitokens); larger values clamp. The v0.2 mutex impl had no such cap. This
  is an enormous burst ceiling for rate limiting and does not affect realistic
  configurations.
- **`retry_after` is reported at millisecond resolution** (the v0.2 impl used
  nanoseconds), and the millisecond counter saturates after ~49.7 days of clock
  advance — call `reset()` periodically on processes that run longer.

## Verification

Run on Windows x86_64 and on Linux x86_64 (WSL2 Ubuntu), Rust stable 1.95.x and
MSRV 1.85; the MSRV build is also enforced by the CI matrix. Every command
passed on both platforms and both toolchains:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo clippy --all-targets --no-default-features -- -D warnings
cargo test --all-features
cargo test --no-default-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
RUSTFLAGS="--cfg loom" cargo test --test loom_acquire
cargo deny check
cargo audit
```

Counts at this tag:

- `--all-features`: 28 unit tests + 1 allocation audit + 3 property tests + 21
  doctests.
- `--no-default-features`: 1 unit test + 2 doctests.
- `loom`: 1 model check.

`cargo deny check` reports advisories, bans, licenses, and sources all ok;
`cargo audit` scans 144 dependencies with no known vulnerabilities.
