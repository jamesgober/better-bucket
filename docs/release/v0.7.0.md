# better-bucket v0.7.0 — Hardening + API Freeze

**Proven under fire, and the surface locked.** v0.7.0 puts the bucket through an
adversarial and edge-case gauntlet, fixes a latent long-uptime stall caught in
the process, and signs off the public API as frozen until 1.0. No signatures
changed.

## What is better-bucket?

A genuinely better token bucket for Rust: lock-free, allocation-free, never
over-grants. Token bucket is its sole job; it is the primitive `rate-net`
consumes.

## What's new in 0.7.0

### An adversarial and edge-case suite

`tests/hardening.rs` adds 13 cases, run by the full CI matrix
(Linux/macOS/Windows × stable + MSRV). Each asserts the safety contract — no
panic, no wrap or overflow, no over-grant, tokens always within `[0, capacity]`:

- **Hostile inputs:** a request of `u32::MAX` (denied forever, no overflow),
  capacity and refill at `u32::MAX` (construct and work, clamped), an extreme
  rate (`u32::MAX` tokens per nanosecond), and a near-zero rate (one token per
  `Duration::MAX`).
- **The edge matrix:** capacity `0` / `1`, requests of `0` / `n-1` / `n` /
  `n+1`, exact-empty and exact-full, zero-time-delta, and a clock that never
  advances.
- **Extreme elapsed:** a ~317-year single advance (clamps, never wraps past
  capacity) and the cross-wrap test below.
- **Concurrent reconfiguration:** six threads acquiring while a seventh resets
  the bucket two thousand times — no panic, no deadlock, always within bounds.

### The long-uptime stall, fixed

Setting up the "enormous elapsed durations" tests surfaced a real latent bug.
The 32-bit millisecond time field used to **saturate** at `u32::MAX`, so after
~49.7 days of uptime an actively-used bucket would **stop refilling entirely** —
and `reset()` could not recover it, because the time anchor is fixed at
construction. For a primitive meant to run for months inside a long-lived
service, that is a latent outage.

v0.7.0 makes the field **wrap** instead, computing elapsed time with
`wrapping_sub`. That is correct for any gap shorter than ~49.7 days — i.e. for
any bucket used at least once in that window, which is every real limiter — so
an actively-used bucket now refills correctly **indefinitely**. The only residual
edge is a bucket left *fully idle* for longer than ~49.7 days, which may
under-refill once on its next use: a safe, conservative, self-correcting outcome.

This is an internal change with no API impact, and every safety invariant still
holds. The proof is a test that advances a live bucket in 40-day steps across the
~49.7-day boundary and asserts it keeps refilling — a test the old saturating
code fails.

### Documentation

`docs/API.md` is complete and accurate to the frozen surface, and the `reset()`
documentation no longer describes a long-uptime workaround it never actually
provided. The README and API limits sections describe the wrap behavior plainly.

## Breaking changes

**None.** The public API is unchanged and now **frozen until 1.0**:

`Bucket` (`per_second`, `per_duration`, `from_config`, `builder`, `with_clock`,
`acquire`, `try_acquire`, `available`, `capacity`, `config`, `reset`),
`BucketBuilder`, `BucketConfig`, `Decision`, `BucketError`, `TokenBucket`, and
`VERSION`. Only additive, non-breaking changes from here.

The time-field change is behavioral, not API: long-uptime refill went from
"stalls after ~49.7 days" to "correct indefinitely for active buckets."

## Verification

Run on Windows x86_64 and on Linux x86_64 (WSL2 Ubuntu), Rust stable 1.95.x and
MSRV 1.85; the MSRV build is also enforced by the CI matrix.

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --no-default-features -- -D warnings
cargo test
cargo test --no-default-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps
RUSTFLAGS="--cfg loom" cargo test --test loom_acquire
cargo deny check
cargo audit
```

Counts at this tag (default features): 34 unit tests + 1 allocation audit + 13
hardening tests + 3 property tests + 27 doctests; `loom`: 1 model check.
`cargo deny check` reports advisories, bans, licenses, and sources all ok;
`cargo audit` is clean.
