# better-bucket v0.5.0 — Feature Complete

**Everything a consumer needs, and the door closed behind it.** v0.5.0 adds the
Tier-2 builder, a full set of runnable examples, and a `criterion` baseline
suite — then declares the public API **frozen** until 1.0. Nothing about the
lock-free core changed; this release rounds out the surface around it and locks
the shape. From here the work is optimization, hardening, and the comparative
benchmark, none of which touch the signatures you build against today.

## What is better-bucket?

A genuinely better token bucket for Rust. One line to construct
(`Bucket::per_second(100)`), one to check (`bucket.try_acquire(1)`), and that
one-line path is the lock-free fast path. Refill is lazy — computed from a
monotonic clock on access, no background timer. The bucket never over-grants,
and that is proven by `loom`, a stress test, an allocation audit, and property
tests. Token bucket is its sole job; it is the primitive `rate-net` consumes.

## What's new in 0.5.0

### `Bucket::builder()` — the Tier-2 path

When `per_second` / `per_duration` are not enough — a capacity and refill rate
that differ, or a non-full initial fill — the builder configures each knob
explicitly and validates the result:

```rust
use better_bucket::Bucket;
use std::time::Duration;

// Burst up to 1000, refill 50/second, start empty.
let bucket = Bucket::builder()
    .capacity(1000)
    .refill(50, Duration::from_secs(1))
    .initial(0)
    .build()?;

assert_eq!(bucket.capacity(), 1000);
assert_eq!(bucket.available(), 0);
# Ok::<(), better_bucket::BucketError>(())
```

`build()` runs the same validation as `BucketConfig::new`, so an unworkable
combination is rejected up front. `initial` defaults to the capacity (the bucket
starts full). For a custom time source, chain the existing `with_clock` onto the
built bucket — the builder stays non-generic and the clock seam is the one that
already exists.

### A full `examples/` set

Four runnable examples, each `cargo run --example <name>`:

- **`per_second`** — the Tier-1 limiter: five allowed, the rest throttled.
- **`burst`** — a bucket absorbing a spike up to capacity, then refilling
  proportionally (driven by a `ManualClock`, so the output is deterministic).
- **`deterministic_test`** — the pattern to copy into your own suite: share a
  `ManualClock`, advance time by hand, assert exact counts, never `sleep`.
- **`builder`** — Tier-2 configuration, including a rejected zero-capacity build.

### Baseline benchmarks

`benches/bucket_bench.rs` now measures the real `Bucket`: single-thread
`try_acquire`, contended acquire across 2/4/8 threads, and the refill
computation after a long idle gap. The numbers are recorded in
[`docs/BENCHMARKS.md`](BENCHMARKS.md). The headline finding, stated honestly:

- The bucket's own accounting — packed-word load, saturating refill, CAS — is
  **~4 ns** (measured with a `ManualClock` to isolate it).
- The single-thread `try_acquire` figure (~26 ns) is **dominated by the
  monotonic clock read** (`Instant::now()`), not the bucket logic.
- Contended throughput scales with threads; the lock-free CAS has no lock to
  serialize on.

The comparative benchmark against `governor` — the one that has to earn the
crate's name — is deferred to `0.6.0`, committed with method and any case it
does not win.

### Refill-after-long-idle, proven

A new test advances the clock five years against a drained bucket and asserts it
saturates to capacity with no wrap or overflow — the long-idle edge that the
saturating math exists to handle.

## Design decisions recorded at this freeze

- **Token bucket is the sole algorithm.** No leaky-bucket or sliding-window view
  is added: they do not share the `TokenBucket` trait cleanly, and per the 1.0
  scope they belong in `rate-net`. This crate owns token-bucket accounting and
  nothing else.
- **The API is frozen until 1.0** — `Bucket`, `BucketBuilder`, `BucketConfig`,
  `Decision`, `BucketError`, `TokenBucket`, and `VERSION`. Only additive,
  non-breaking changes from here.

## Breaking changes

**None.** `BucketBuilder` and `Bucket::builder()` are purely additive; every
existing signature is unchanged.

## Verification

Run on Windows x86_64 and on Linux x86_64 (WSL2 Ubuntu), Rust stable 1.95.x and
MSRV 1.85; the MSRV build is also enforced by the CI matrix. Every command
passed on both platforms and both toolchains:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets --all-features -- -D warnings
cargo clippy --all-targets --no-default-features -- -D warnings
cargo test --all-features
cargo test --no-default-features
cargo run --example per_second   # and burst, deterministic_test, builder
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
RUSTFLAGS="--cfg loom" cargo test --test loom_acquire
cargo bench --bench bucket_bench
cargo deny check
cargo audit
```

Counts at this tag:

- `--all-features`: 34 unit tests + 1 allocation audit + 3 property tests + 27
  doctests.
- `--no-default-features`: 1 unit test + 2 doctests.
- `loom`: 1 model check.

`cargo deny check` reports advisories, bans, licenses, and sources all ok;
`cargo audit` scans 144 dependencies with no known vulnerabilities.
