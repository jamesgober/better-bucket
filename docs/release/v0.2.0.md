# better-bucket v0.2.0 — Foundation

**The public surface, locked.** v0.2.0 turns the scaffold into a working token
bucket: `Bucket`, `BucketConfig`, `Decision`, `BucketError`, and the
`TokenBucket` trait are all in place, documented, and exercised by a property
suite that encodes the crate's safety contract. The implementation underneath is
deliberately simple — a mutex around millitoken accounting — so the API and the
invariants can be settled *before* the hard part. The lock-free, allocation-free,
cache-aligned core that earns the crate its name replaces those internals in
v0.3.0 behind the exact same surface.

## What is better-bucket?

A genuinely better token bucket for Rust. The common case is one line —
`Bucket::per_second(100)` then `bucket.try_acquire(1)` — and that one-line path
is designed to be the fast path. Refill is lazy: tokens accrue from a monotonic
clock the moment you ask, with no background timer thread. The defining
correctness property is that the bucket never over-grants — across any sequence
of acquires and time advances, total tokens handed out never exceed the initial
fill plus accrued refill. It is the single-purpose home for token-bucket math in
the wider ecosystem and the primitive the `rate-net` rate limiter consumes.

## What's new in 0.2.0

### `Bucket` — the token bucket

The headline type, generic over its time source (`Bucket<C: Clock = SystemClock>`):

```rust
use better_bucket::Bucket;

// 100 tokens/second, capacity 100, starting full.
let bucket = Bucket::per_second(100);

if bucket.try_acquire(1) {
    // allowed
}
```

Three constructors cover the tiers of need:

- `Bucket::per_second(rate)` and `Bucket::per_duration(amount, period)` — the
  infallible Tier-1 path. A zero rate yields a bucket that grants nothing
  (a well-defined deny-all), so the common case never has to handle a `Result`.
- `Bucket::from_config(config)` — full control via a validated `BucketConfig`,
  for buckets where capacity, rate, and initial fill differ (e.g. a large burst
  ceiling with a slow refill, or one that starts empty).

The acquire surface is two methods. `try_acquire(n) -> bool` is the one-line
convenience; `acquire(n) -> Decision` returns the full outcome, including how
long to wait when denied. Both are infallible — the bucket never returns a
`Result` on the hot path. `available()`, `capacity()`, and `config()` round out
the inherent surface.

### `BucketConfig` — validated parameters

```rust
use better_bucket::{Bucket, BucketConfig};
use std::time::Duration;

// Burst up to 500, refill 100/sec, start empty.
let config = BucketConfig::new(500, 100, Duration::from_secs(1), 0)?;
let bucket = Bucket::from_config(config);
assert_eq!(bucket.available(), 0);
# Ok::<(), better_bucket::BucketError>(())
```

`BucketConfig::new` rejects configurations that cannot describe a working bucket
— zero capacity, zero refill amount, or zero refill period — returning a
`BucketError` up front. `initial` is clamped to `capacity`. An invalid bucket
cannot be constructed.

### `Decision` and `BucketError`

`Decision` is the acquire outcome: `Allowed`, or `Denied { retry_after }` where
`retry_after` is the minimum wait until the same request would succeed
(`Duration::MAX` when the request exceeds capacity and can never succeed). It is
exactly what a downstream limiter needs to populate a `Retry-After`.

`BucketError` is the `#[non_exhaustive]` construction error. It is hand-written
so every variant is documented, and it implements `error_forge::ForgeError` to
slot into the portfolio error stack — `kind()` returns the variant name,
`is_retryable()` is `false` (a bad config will not fix itself).

### `TokenBucket` trait

```rust
use better_bucket::{Bucket, TokenBucket};

fn drain(bucket: &dyn TokenBucket) -> u32 {
    let mut taken = 0;
    while bucket.try_acquire(1) {
        taken += 1;
    }
    taken
}

assert_eq!(drain(&Bucket::per_second(5)), 5);
```

The object-safe abstraction a consumer codes against, implemented for every
`Bucket<C>`. This is the surface `rate-net` will hold so it can use the bucket
without naming its concrete clock type.

### Deterministic time via clock injection

`with_clock` swaps the bucket's time source. Injecting a `ManualClock` from
`clock-lib` makes refill behaviour fully deterministic in tests — advance the
clock by hand, assert the token count, never `sleep`:

```rust
use better_bucket::Bucket;
use clock_lib::ManualClock;
use std::sync::Arc;
use std::time::Duration;

let clock = Arc::new(ManualClock::new());
let bucket = Bucket::per_second(10).with_clock(Arc::clone(&clock));

assert!(bucket.try_acquire(10));     // drain
clock.advance(Duration::from_millis(500)); // half a period
assert_eq!(bucket.available(), 5);   // refilled proportionally
```

### Property-based safety suite

`tests/invariants.rs` encodes the two contracts with `proptest`:

- **Tokens stay within `[0, capacity]`** across arbitrary sequences of acquires
  and time advances.
- **Never over-grant** — with no refill, total grants never exceed the initial
  fill; with refill, they never exceed the initial fill plus what the configured
  rate could accrue over the elapsed time.

These are the same invariants the lock-free v0.3.0 core must continue to
satisfy, so the suite carries forward unchanged.

## Breaking changes

**None at the public-API level** — there was no token-bucket API in 0.1.0 to
break. One packaging change worth noting: the `clock` feature now implies `std`
and gates the `Bucket` surface (clock-lib's `Clock` trait is std-gated, and the
simple implementation uses a `Mutex`). A bare `no_std` build exposes only
`VERSION` until the caller-driven core lands in 0.3.0.

## What's next

- **0.3.0 — The lock-free core.** Replace the mutex-backed internals with a
  packed atomic `(tokens, last_refill_tick)` word and a `compare_exchange_weak`
  acquire loop: zero allocation on the hot path, cache-line padding against false
  sharing, a `loom` model of the acquire/refill interleaving proving no
  over-grant, and a multi-thread stress test. The public surface from this
  release does not change.
